  # Code as Written Knowledge

  ### A different way to think about software development

  .footnote[Kuhn Tech Tails — 2026]

  ---

  # The usual framing

  We talk about software development as *building* things.

  - Architects design systems
  - Engineers construct features
  - QA tests the structure

  > "We're building a platform."

  ---

  # But buildings don't change their meaning

  A bridge built in 1950 means the same thing today.

  Code written in 2020 may mean something *completely different* by 2025 —
  because the problem it solves has changed.

  ---

  # What if we thought of code as *writing*?

  - Authors draft, revise, and collaborate
  - Readers need to understand intent, not just structure
  - Meaning evolves with context

  > "We're authoring a codebase."

  ---

  # What changes with this framing

  | Building | Writing |
  |----------|---------|
  | Blueprints | Docs & comments |
  | Load-bearing walls | Core abstractions |
  | Renovation | Refactoring |
  | Demolition | Deletion |

  Both metaphors hold — but *writing* puts **clarity** first.

  ---

  # Clarity as a first-class concern

  When you treat code as writing, the question shifts from:

  > "Does it work?"

  to:

  > "Will the next person understand *why* it works?"

  ---

  # Practical takeaways

  1. Name things for the reader, not the compiler
  2. Delete code that no longer communicates anything useful
  3. A short, clear function is better than a clever long one
  4. Comments explain *why*, not *what*
